This is an document containing examples for correctly parsed course requirements.

The format is a JSON object, with a field 'prerequisites' and 'corequisites'.
The values for these have a tree-like structure, where each node is either a course code or a grouping of nodes.
A course code is simply a 4 character string (e.g. 'COMP') followed by a 3 digit number (e.g. '202'), with potentially 'D1', 'D2', 'J1', 'J2', or 'J3' appended to the 3 digit number.
Examples of valid course codes are 'COMP 250', 'MATH 240', 'CMR2 564', 'COMP 361D1', and 'DENT 101J3'.

Requirements that are not course codes or courses that are not strictly required should not be included in the output
Some examples of strings that shouldn't be included are 'instructor permission' and 'equivalent'.

Lets call this type `ReqNode`, and the type is defined like so:

`type ReqNode = string | {operator: 'AND' | 'OR', groups: ReqNode[]}`

Let's also call the type of the output format `CourseReq`, and it is defined like so:

`type CourseReq = {prerequisites: ReqNode | null, corequisites: ReqNode | null}`

A group of course codes has an operator, which is either 'AND' or 'OR', which indicates the relation between the courses as requirements.
The groups are then a list of `ReqNode`s, which can be either course codes or other groups.
`ReqNode`s that are strings MUST be a course code. Course codes in the input text will be surrounded by backticks, so they can be easily identified.
Do not include `ReqNode` string values that are not course codes in the output.

If there are no prerequisites or corequisites, the value for the field is `null`.

Here are examples for expected results when parsing course requirements.
The input text is preceded by ---INPUT---, and the expected result is preceded by ---OUTPUT---.

Prerequisite: `MATH 141`. Familiarity with vector geometry or Corequisite: `MATH 133`
---OUTPUT---
{
  "prerequisites": "MATH 141"
  "corequisites": "MATH 133"
}
